# V2.2 — Tool Execution Layer for OpenRouter/API Backends

## Implementation Plan

---

## 1. 架构概览与数据流

当前 OpenRouter adapter 是单次调用：发送 `messages`，获取回复，返回 `BackendResponse.text`。`supports_tool_use=False`。

工具循环在 **adapter 内部** 引入多轮对话：

```
用户 prompt
    ↓
OpenRouter API 调用（带 tools 参数）
    ↓
finish_reason == "tool_calls" ?
    YES → Bridge 本地执行工具
        → 追加 assistant message + tool_result 到 messages
        → 重新调用 API（循环）
    NO  → 返回最终文字作为 BackendResponse
```

**关键设计决策**：工具循环住在 adapter 内部（`OpenRouterAdapter`），不改 runtime 的主循环。Runtime 看到的仍然是普通的 `BackendResponse`。adapter 在执行工具时通过 `StreamEvent` 回调展示实时活动。

### D. `tools/registry.py`
工具注册表与权限检查器：

```python
class ToolRegistry:
    def __init__(self, allowed_tools: list[str], access_root: Path, workspace_dir: Path, secrets: dict)
    def get_tool_definitions(self) -> list[dict]        # 返回允许的工具 schema
    async def execute(self, tool_name, arguments) -> ToolResult   # 执行并强制权限
    def is_allowed(self, tool_name) -> bool

@dataclass
class ToolResult:
    tool_call_id: str
    output: str
    is_error: bool = False
```

## 4. 工具循环机制（详细）

改动核心在 `OpenRouterAdapter.generate_response()`。

### 当前行为
构建 2 条消息（system + user），单次 API 调用。

### 新行为

```python
async def generate_response(self, prompt, request_id, ...) -> BackendResponse:
    messages = [
        {"role": "system", "content": self.sys_prompt},
        {"role": "user", "content": prompt}
    ]

    for loop_idx in range(MAX_TOOL_LOOPS):  # 默认 25
        payload = self._build_payload(messages)

        if use_streaming:
            result = await self._stream_response_with_tools(payload, headers, ...)
        else:
            result = await self._call_api(payload, headers)

        if not result.tool_calls:
            return BackendResponse(text=result.text, ...)

        # 追加 assistant 消息（带 tool_calls）
        messages.append({
            "role": "assistant",
            "content": result.text,
            "tool_calls": result.tool_calls
        })

        # 逐个执行工具，追加结果
        for tc in result.tool_calls:
            # 发出 stream event
            await self._emit_tool_event(KIND_TOOL_START, tc)

            tool_result = await self.tool_registry.execute(
                tc["function"]["name"],
                json.loads(tc["function"]["arguments"])
            )

            messages.append({
                "role": "tool",
                "tool_call_id": tc["id"],
                "content": tool_result.output
            })

            await self._emit_tool_event(KIND_TOOL_END, tc, tool_result)

    # 超出最大循环次数
    return BackendResponse(text=accumulated_text or "Tool loop limit reached", ...)
```

### Streaming 中的 tool_calls 累积

OpenRouter streaming 返回 tool_calls 的方式：
```json
{"choices": [{"delta": {"tool_calls": [{"index": 0, "id": "call_xxx", "function": {"name": "bash", "arguments": ""}}]}}]}
{"choices": [{"delta": {"tool_calls": [{"index": 0, "function": {"arguments": "{\"com"}}]}}]}
{"choices": [{"delta": {"tool_calls": [{"index": 0, "function": {"arguments": "mand\": \"ls\"}"}}]}}]}
```

需要在 streaming 路径中累积这些 delta 片段，拼装成完整的 tool_call 对象。

### 内部中间结果

```python
@dataclass
class _APIResult:
    text: str
    tool_calls: list[dict] | None  # None = 纯文字回复
    finish_reason: str
```

## 5. 权限与安全模型

### agents.json 扩展

在 `allowed_backends` 条目中添加 `tools` 键：

```json
{
  "engine": "openrouter-api",
  "model": "anthropic/claude-sonnet-4.6",
  "tools": {
    "allowed": ["bash", "file_read", "file_write", "web_search", "web_fetch"],
    "max_loops": 15,
    "bash": {
      "timeout_max": 60,
      "blocked_patterns": ["rm -rf /", "sudo rm"]
    },
    "file_write": {
      "max_file_size_kb": 500
    }
  }
}
```

- `tools` 键缺失 → 无工具（向后兼容）
- `allowed: ["*"]` → 全部工具
- 每个工具可有独立的安全配置

### 路径沙箱

`file_read` 和 `file_write` 将路径解析到 `access_root`（来自 `AgentConfig`）。任何路径穿越（`..`）超出 access_root 的都会被拒绝。

### bash 安全

- `cwd` = `workspace_dir`
- 超时限制（默认 30s，可配置最大值）
- 可选的 `blocked_patterns` 列表

### API Key

Brave Search API key 从 `secrets.json` 读取：`"brave_api_key"`

## 7. 错误处理

| 场景 | 处理方式 |
|------|---------|
| 工具执行失败 | 返回错误字符串作为 tool result（`is_error=True`），模型可看到并重试 |
| 工具不在允许列表 | 返回 `"Error: tool 'X' is not in your allowed tools list"` |
| 路径越界 | 返回 `"Error: path is outside allowed access scope"` |
| bash 超时 | kill 进程，返回 `"Error: command timed out after Xs"` |
| 超过最大循环次数 | 中断循环，返回已累积的文字 + 警告 |
| API 错误（循环中） | 立即返回 `BackendResponse(is_success=False, error=...)` |
| JSON 解析失败 | 捕获 `JSONDecodeError`，返回错误给模型 |

## 9. 实施顺序

### Step 1：`tools/schemas.py`
纯数据，无依赖。定义 5 个工具的 JSON Schema。

### Step 2：`tools/builtins.py`
实现五个工具执行器。每个是独立的 async 函数。
- `execute_bash` — `asyncio.create_subprocess_shell`，超时，输出截断
- `execute_file_read` — `aiofiles` 或同步 `open()`，offset/limit
- `execute_file_write` — 写入文件，父目录自动创建
- `execute_web_search` — `httpx` 调用 Brave Search API
- `execute_web_fetch` — `httpx` 获取 HTML，`html2text` 转 Markdown

### Step 3：`tools/registry.py`
`ToolRegistry` 类，加载 schemas，权限检查，调度执行。

### Step 4：`tools/__init__.py`
导出 `ToolRegistry`、`ToolResult`。

### Step 5：修改 `adapters/openrouter_api.py`（核心改动）
- 添加 `tool_registry` 属性
- 新增 `_build_payload(messages)` 方法
- 重构 `generate_response()` 为工具循环
- 重构 `_stream_response()` 以累积 tool_calls delta
- 新增 `_execute_tool_calls()` 方法
- 当 tools 存在时 `supports_tool_use=True`

### Step 6：修改 `adapters/base.py`
`BackendResponse` 添加可选的 `tool_calls` 和 `stop_reason` 字段（为未来扩展准备）。

### Step 7：修改 `orchestrator/flexible_backend_manager.py`
在 `initialize_active_backend()` 中，创建 backend 后，如果 backend config 包含 `tools` 键，创建 `ToolRegistry` 并注入 adapter。

### Step 8：更新 `agents.json`
为需要工具的 OpenRouter backend 条目添加 `tools` 配置。

### Step 9：测试与验证
